# WMS Monitor — 仓储拣货实时监控系统

> 面向大型生鲜仓储中心的实时监控平台，支持多仓库并行监控，覆盖拣货进度追踪、人效分析、卡单预警、异常包裹巡检、出勤管理、权限自动化等核心场景。系统采用 **Flask + APScheduler + SQLite** 架构，通过多层数据采集管线将 WMS 内部接口数据实时同步至本地，以 ECharts 可视化看板对外呈现。每个仓库使用独立的 SQLite 数据库文件实现完全的数据隔离，仓库切换由前端 localStorage 管理，服务端完全无状态，支持多用户同时查看不同仓库的数据互不干扰。

## 项目背景

在日均处理数万件商品的生鲜仓配中心，拣货环节的效率直接影响次日履约质量。传统 WMS 系统仅提供基础数据查询，缺乏实时聚合、异常预警和人效洞察。本项目从零搭建了一套独立的监控层，在不侵入原有 WMS 的前提下，通过 API 数据采集 + 本地计算 + 可视化看板的方式，实现全链路实时监控。当前已接入深圳凤岗仓（428）和深圳清溪仓（636），新增仓库只需在配置中添加一行即可自动纳入采集和展示。

## 核心功能

**实时拣货监控** — 以 30 秒为周期从 WMS 接口拉取全仓拣货进度，聚合计算每位拣货员的实时拣货量、分区效率、空闲时长，并通过 10 分钟粒度快照生成同日同时段对比曲线（Sparkline），帮助现场主管实时掌握拣货节奏与历史偏差。

**卡单智能检测** — 双维度卡单判定算法：①「拣完未提交」检测（实拣量已达应拣量但状态超 3 分钟未变更）；②「未完成停滞」检测（实拣量未达标且操作时间超 3 分钟未更新）。检测结果实时入库并推送告警，显著缩短卡单发现到处理的响应时间。

**人效分析与日报** — 自动计算每位拣货员的投线区/调料区拣货效率、计件/计时标签、新员工标识（入职 7 天内），在全部拣货任务完成后自动生成当日拣货总结报表，支持 CSV 导出和跨日对比分析。

**异常包裹巡检** — 跨 9 个目标仓库定时扫描包裹数量异常明细，通过本地防重缓存池 + 逻辑熔断机制避免重复告警，自动关联 SKU 品牌信息并标记处理状态。

**出勤甘特图与班组管理** — 按小时粒度统计各班组出勤人数，渲染全天出勤甘特图，支持自定义人员-班组映射配置，覆盖临时工/正式工混合排班场景。

**权限自动补开** — 每日开工后自动检查所有活跃拣货人员的系统权限（共享拣货扫码 GXJHSM、RF 操作员 RFCKCZ），对缺失权限的员工自动调用权限分配接口补开，避免因权限问题导致的拣货中断。

**销售预测集成** — 通过浏览器自动化从 BI 看板抓取 TN 算法预测值，每日 19:05 首次触发，之后每 30 分钟检测补抓，将次日预测件量入库供进度对比参考。

**标签打印** — 基于 Playwright 渲染 HTML/CSS + JsBarcode + QRCode 生成高精度标签图片（203dpi），通过 win32print 调用热敏打印机输出任务汇总签和包裹标签，渲染质量与官方打印工具一致。

## 技术亮点

**多级鉴权降级机制** — 设计了四层鉴权降级链路（CatDesk CLI → mtsso 本地交换 → Token 本地文件缓存 → Cookie 直连），当企业 SSO 短暂不可用时自动降级，确保监控系统在鉴权服务波动时仍能持续运行。Token 获取使用线程锁保护，缓存有效期 57 分钟，到期自动刷新。

**并发数据采集与数据库并发控制** — 拣货数据采集使用 `ThreadPoolExecutor` 并发拉取多人员考勤信息，同时通过 `threading.Lock` 防止多轮采集重叠。数据库层启用 SQLite WAL 模式提升并发读写能力，并实现 `execute_with_retry` 包装器在遇到 `database is locked` 时自动指数退避重试。

**多仓库数据隔离** — 每个仓库使用独立的 SQLite 数据库文件（如 `wms_monitor.db`、`wms_monitor_636.db`），通过 `get_db(warehouse_id)` 动态路由到对应文件，从根本上避免跨仓库数据污染。仓库选择完全由前端 localStorage 管理，API 通过请求参数 `warehouseId` 确定目标仓库，服务端不持有任何仓库状态，多用户可同时查看不同仓库互不影响。新增仓库只需在 `WAREHOUSES` 字典中添加一行配置，系统会自动创建数据库文件并纳入采集循环。

**按需调度与智能生命周期管理** — 调度器不是简单的定时轮询，而是实现了完整的生命周期管理：每轮轮询自动遍历所有已配置仓库执行数据采集、用户访问时激活、离开后 10 分钟超时自动暂停、18:00 自动激活进入工作时段模式（持续到凌晨 1:30）、全部拣货完成后自动停止并生成总结、服务重启时自动回填缺失时段快照。

**逻辑日与波次感知** — 精确处理夜班场景下的逻辑日期计算（8 点前归属前一日），并按履约波次（上午达/下午达/凌晨达）拆分拣货进度快照，支持按波次维度的独立进度追踪和效率分析。

**双模式标签渲染引擎** — 使用 Playwright 无头浏览器渲染 HTML/CSS 模板生成标签图片，结合 JsBarcode（条形码）和 QRCode（二维码），通过精确的 DPI 换算（203dpi → 像素映射）输出到热敏打印机，兼顾渲染灵活性和打印精度。

## 快速开始

```bash
# 安装依赖
pip install flask flask-cors apscheduler requests

# 标签打印（可选）
pip install playwright Pillow
playwright install chromium

# 启动服务
python app.py
# 服务运行在 http://localhost:5000
```

访问 `http://localhost:5000` 进入功能门户，在顶部切换目标仓库（默认深圳凤岗仓 428），选择对应功能模块进入监控看板。系统会在工作时段（18:00 ~ 01:30）自动激活数据采集并遍历所有已配置仓库，也可在页面中手动触发。

## 多仓库配置

在 `spiders/base.py` 中的 `WAREHOUSES` 字典添加新仓库即可，系统启动时会自动创建对应的数据库文件：

```python
WAREHOUSES = {
    "428": {"name": "深圳凤岗仓", "short_name": "凤岗"},
    "636": {"name": "深圳清溪仓", "short_name": "清溪"},
    # 新增仓库只需加一行:
    # "999": {"name": "新仓库名称", "short_name": "简称"},
}
```

数据库文件命名规则：默认仓库(428)使用 `wms_monitor.db`，其他仓库使用 `wms_monitor_{仓库ID}.db`。各仓库的实时状态、历史快照、拣货数据完全隔离，考勤和权限数据在主仓库数据库中共享。
